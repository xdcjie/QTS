QTS Backtest 完整链路图

从配置到加载、读历史数据、连续合约换约、数据视图、策略处理、风险检查、订单执行、账户更新和报告输出。 这张图按当前实现描述，同时标出 bar aggregation 与共享运行时的目标边界。

核心原则

live 通过 gateway/adapters 指定实时数据从哪里来；backtest 通过 market_data.config 和 catalog 指定历史数据从哪里来。 数据源配置是项目级，策略配置是策略实例级，二者不应混在一起。

1. 配置责任边界

回测配置只引用数据源和策略实例；项目级历史数据配置负责物理目录和 catalog；策略配置只描述策略类和参数。

Backtest Run Config

一次回测场景

选择数据源 catalog、回测时间段、timeframe、初始资金、成本模型、风险参数、roll policy 和 warmup。 它不拥有历史数据目录细节，也不拥有策略内部参数。

Historical Data Config

项目级数据源

定义本项目有哪些历史数据 store、每个 store 的根目录、bars 目录、chain 目录、文件模板、时区策略和 normalization。 data_dir / chain_dir 属于 store，不属于 GC/SI 这种商品条目。

Catalog / Dataset

逻辑数据集

catalog 说明这次可用的数据集合，例如 research_futures。 dataset 只描述 GC、SI 的资产类型、交易所和可选文件覆盖项。

Strategy Config

策略实例

描述策略 ID、class path、account、allocation、enabled 和策略自己的 params。 策略参数不应该知道历史数据目录，也不应该直接接触 broker / account / risk core。

2. 要配置什么，每个配置什么意思

三类配置共同组成一次回测：run manifest 决定“怎么跑”，historical data config 决定“数据从哪里来”，strategy config 决定“跑哪个策略以及策略参数”。

Backtest Run Manifest

一次回测任务的入口配置，由 BacktestRunConfig.from_yaml 读取。

market_data.source 数据源类型。当前示例是 local_historical，语义等价于 live 中的 gateway/adapters 类型。
market_data.config 项目级历史数据配置文件路径，例如 configs/data/historical.local.yaml。
market_data.catalog 从历史数据配置中选择哪个 catalog，例如 research_futures。
roots 需要加载的商品根，例如 GC、SI。runner 用它解析 CSV 和 chain。
symbols 策略可见/回测关注的符号集合。启用换约时，GC、SI 可作为连续期货 research symbol。
start / end 回测时间窗口，按半开区间 [start, end) 过滤历史 bars。
timeframe 策略请求的 bar 周期。它和历史源 source_timeframe 分离；二者不同时由 MarketDataActor / BarAggregator 生成策略周期。
initial_cash 账户初始资金，用于初始化 AccountActor。
strategy_config 策略实例配置文件路径。旧字段 strategy_class / strategy_params 仍可兼容。
cost_model 回测成交成本，例如每张固定佣金、滑点 bps，由回测执行适配器使用。
risk_config 风险规则配置，例如 max_notional，由 RiskEngine 检查。
roll_policy 连续期货换约规则，例如 enabled: true、method: highest_volume。
warmup_bars 预热 bar 数。策略可接收数据更新指标，但 warmup 期间新增交易意图不会下单。
configs/data/historical.local.yaml

Historical Data Source

项目级数据源配置，由 HistoricalDataConfig.from_yaml 读取。

stores.local_csv.type 物理 store 类型。当前是 local_csv，表示从本地 CSV/JSON 读取。
root_dir 历史数据根目录，例如 historical。
bars_dir bar CSV 子目录，例如 data，最终路径会拼成 historical/data/gc.csv。
chains_dir 期货 chain JSON 子目录，例如 chains，最终路径会拼成 historical/chains/GC.json。
bars_file_template bar 文件命名模板，例如 {root_lower}.csv。
chain_file_template chain 文件命名模板，例如 {root}.json。
source_timeframe 物理 CSV 数据源的 bar 周期，例如 1m。这是数据源能力，不是策略请求。
exchange_timezone 聚合 bar 时使用的交易所时区，例如 US/Eastern。用于 clock-aligned timeframe 边界。
timezone_policy 源数据时间解释策略。当前示例为 source_utc_exchange_sessions。
normalization 价格/成交量 normalization 策略。当前示例为 raw。
catalogs.*.store catalog 使用哪个 store，例如 research_futures 使用 local_csv。
datasets.GC / SI 逻辑商品条目，描述资产类型、交易所和可选文件覆盖项；不放公共数据目录。
configs/strategies/gc_si_momentum.yaml

Strategy Instance

策略实例配置，由 runner 加载后实例化策略类。

strategy_id 策略实例 ID，用于日志、报告和后续多策略区分。
class_path 策略类导入路径，例如 examples.strategies.gc_si_momentum:GcSiMomentumStrategy。
account_id 策略绑定账户。当前 research backtest 用它关联初始账户。
allocation 策略资金分配比例，为后续多策略/多账户管理保留边界。
enabled 是否启用该策略实例。
params 策略自己的参数，例如关注哪些 symbols。这里不放数据目录、chain 目录或风险核心配置。

3. 完整回测链路

每一行是一段职责边界。箭头表示主数据/控制流；actor 内部仍通过 mailbox/message 传递事件，不由策略直接改账户或订单状态。

启动与配置 入口、解析、装配
1 CLI 启动回测 scripts/run_research_backtest.py

读取 --config 和 --output-dir，调用 run_research_backtest。

2 加载 Run Config BacktestRunConfig.from_yaml

解析 market data 引用、回测窗口、timeframe、资金、成本、风险、roll policy、strategy config。

3 加载项目级数据源 HistoricalDataConfig.from_yaml

读取 stores/catalogs/datasets，校验目录职责不下沉到 GC/SI 商品条目。

4 加载策略实例 strategy_config -> class_path

导入策略类，用 params 创建策略对象；策略只拿自己的参数，不拿 runtime 内部对象。

历史数据与换约 source、catalog、bar、roll
5 解析 Catalog load_historical_catalog_from_config

把 research_futures + GC/SI 解析成具体 CSV 路径、chain JSON 路径和 symbol resolver。

6 读取历史 Bar iter_historical_bars

逐行读取 CSV，构造已完成的 Bar，按 [start, end) 时间窗口过滤，统计 rows/bars/spreads。

7 连续合约换约选择 HighestVolumeFutureContractSelector

启用 roll 时，按 timestamp 聚合同一 root 的候选合约，过滤 spread，选择成交量最高的具体合约。

8 登记 Roll Selection FutureRollRegistry

记录连续 symbol 在每个时间点对应的具体合约和执行价格，用于后续意图转订单与换约平旧仓。

9 准备 Source Events BacktestMarketDataEvent

runner 保留 source timeframe bars；排序和 replay 驱动交给 BacktestEngine runtime。

策略数据流 portal、StrategyActor、context
10 装配 BacktestEngine BacktestEngine.from_config

用配置创建成本模型、风险引擎、合约乘数、InstrumentRegistry、MarketDataActor、StrategyActor、SignalAggregatorActor 和执行 actors。

11 推进回测时钟 ReplayClock -> BacktestEngine.run

ReplayClock 按 source event 时间推进；engine 将 source bars 发送给 MarketDataActor。

12 MarketDataActor / BarAggregator MarketDataEvent -> completed Bar

runtime actor 持有 aggregation state；只把 completed requested timeframe bars fan-out 给策略链路。

13 StrategyActor StrategyBarEvent -> StrategyBarResult

Actor 拥有 strategy 和 StrategyContext，收到 completed bar 后用 DataView 更新 data、portfolio、indicator。

14 策略处理 Bar StrategyActor calls strategy.on_bar

策略基于 DataView 和 PortfolioView 产生 TargetIntent；StrategyActor 只返回本 bar 新产生的 intents。

意图、风控与执行 shared business path
15 Warmup Gate warmup_bars

预热期仍更新数据和指标，但新增交易意图不进入下单链路。

16 SignalAggregatorActor StrategySignalEvent -> AggregatedSignalBatch

非预热 bar 的 strategy intents 先进入信号聚合边界；当前单策略为 pass-through，后续可承载多策略合并。

17 意图解析到可交易合约 continuous future -> concrete contract

连续合约 research symbol 会通过 roll registry 转成具体合约；若换约，先平旧相关合约，再开新目标仓位。

18 风险检查 RiskEngine.check

用价格、数量、乘数和账户状态检查 MaxNotionalRule 等规则；拒绝则不产生订单。

19 订单状态管理 SubmitOrder -> OrderManagerActor

订单进入统一状态机，由 OrderManagerActor 拥有订单状态；策略不直接改订单。

20 回测成交适配器 ExecutionActor -> BacktestExecutionAdapter

按 bar/roll execution price、滑点和佣金模拟立即成交，生成 ExecutionReport。

21 账户更新 OrderFill -> AccountActor

AccountActor 独占账户状态，应用 fill 后更新现金、持仓、成本和 realized PnL。

输出 ledger、equity、report
22 交易流水 TradeLedger

从 fills 生成交易记录，保留 instrument、side、qty、price、commission、order id 等。

23 权益曲线 equity_curve

每个 bar 后按最新价格和乘数 mark-to-market，记录 cash、positions value、equity。

24 指标与哈希 metrics / config_hash / report_hash

计算 total return、max drawdown 等，同时记录配置和报告哈希，支撑可追溯。

25 写入结果文件 report.json / summary.json

CLI 将完整报告和摘要写到 output directory，供后续分析或前端展示。

4. 数据流转与 Bar Aggregation

这里必须区分当前 research backtest 已实现路径和目标共享 runtime 路径。

当前 research backtest 路径

CSV Source Bar 源文件是 source timeframe OHLCV bar，例如 1m。
iter_historical_bars 按 source_timeframe 构造 complete source bars。
MarketDataActor 历史 bars 先进入 runtime market data actor。
BarAggregator source timeframe 和 requested timeframe 不同时聚合。
DataView 策略只看到 actor 输出的 completed requested bars。

当前已经收敛到目标 runtime 形态：research runner 只加载 source bars 和 metadata； BacktestEngine 用 ReplayClock 驱动 source bars 进入 MarketDataActor，由 BarAggregator 输出 completed strategy bars， 再通过 StrategyActor 调用策略，并把非预热 intents 送入 SignalAggregatorActor 后进入订单链路。

目标共享 runtime 路径

Source Events tick、quote、trade 或 source timeframe bars。
MarketDataActor 统一接入 live/paper/backtest 数据源。
BarAggregator 按 instrument、timeframe、exchange session 聚合。
Completed Bar 只发布完整 bar，遵守 [start, end)。
StrategyActor 策略收到统一格式 bar，路径与 live/paper 对齐。
SignalAggregatorActor 合并策略 signals，输出进入风控的目标批次。

这条路径是 backtest/live parity 的目标：差异只在数据源、执行适配器和时钟，market data、strategy、signal、风控、订单、账户链路共享。

5. 连续期货换约链路

连续合约是研究/数据引用，不是最终可交易合约。下单前必须解析到具体合约。

Roll Policy: highest_volume

按 root 加载 roots: GC/SI 决定读取哪些商品历史文件和 chain。
同时间分组 同一 timestamp 下收集可交易候选合约，排除 spread/非目标数据。
选择主力 HighestVolumeFutureContractSelector 选择成交量最高合约。
发连续 Bar 策略看到的是 root/continuous symbol 对应的研究 bar。
下单前解析 意图转订单时解析为具体合约，并处理旧合约平仓。

这能保证策略逻辑可以基于 GC / SI 这种连续研究符号运行， 但订单、持仓、成交和账户永远落在具体合约上。

6. 当前实现 vs 目标架构

这部分用来防止误解：当前 research backtest 已经按 actor + market data runtime 主链路收敛，剩余主要是工程化抽取和更丰富的多策略语义。

主题
当前实现
状态 / 剩余边界
数据源配置

market_data.config 指向项目级 historical_data，catalog 决定使用的数据集合。

保持该边界，并让 live/paper/backtest 都用“source/adapters + catalog/universe”的同构模型。

历史目录归属

root_dir、bars_dir、chains_dir 位于 store 级别，不在 GC/SI 下。

继续禁止把公共物理目录下沉到商品条目；商品条目只保留逻辑元数据和必要覆盖项。

Bar Aggregation

BacktestEngine 已用 ReplayClock 驱动 historical source bars 进入 MarketDataActor；当 source/requested timeframe 不同时由 BarAggregator 输出 completed bars。

已达成目标链路。后续只剩工程化拆分：可把 engine 内部 orchestration 提取成复用 runtime service。

策略处理

StrategyActor 拥有 strategy 和 StrategyContext，通过 StrategyBarEvent 处理 completed bars 并返回新 intents；非预热 intents 进入 SignalAggregatorActor。

已达成 actor 边界；当前 signal aggregation 是单策略 pass-through，后续多策略组合可在该边界扩展。

订单/账户

聚合后的意图通过 RiskEngine、OrderManagerActor、ExecutionActor、AccountActor 形成统一链路。

live/paper/backtest 共享 business path，只替换执行适配器、数据适配器和 clock。

7. 当前推荐 YAML 形态

关键是 backtest 配置只引用数据源和策略配置；历史数据目录在项目级 store 下；策略参数独立。

设计判断：backtest 的 market_data 应像 live 的 gateway 一样表达“数据从哪里来”。 对本地历史回测来说，这个来源不是网络地址，而是项目级 historical_data store/catalog。 策略配置只表达策略实例，不表达历史数据目录。
