# Repository hygiene and lifecycle policy

This document defines how QTS keeps source code, documentation, tests, generated output, and historical evidence separated. The goal is to reduce repository drift without deleting useful history.

## Source of truth

`AGENTS.md` is the canonical instruction file for AI/code agents. Model-specific files such as `CLAUDE.md` and `GEMINI.md` should be short adapters that point back to `AGENTS.md` and only document model-specific deviations.

When two documents disagree, prefer this order:

1. Current source code and tests.
2. `README.md` for user-level behavior.
3. `AGENTS.md` for agent workflow rules.
4. Current documents under `docs/`.
5. Archived or historical documents.

## Generated files

Large generated files or generated inventories should either be excluded from git or marked clearly:

```markdown
<!-- AUTO-GENERATED. DO NOT EDIT BY HAND. -->
Generated by: <script or command>
Source commit: <sha>
```

If a generated document is useful for review but not necessary for the source tree, put it under local output such as `artifacts/` and keep it ignored. If it is required as a stable fixture, put a minimal version under `tests/fixtures/`.

## Runtime output and artifacts

Default location for local outputs:

```text
.local/
artifacts/
runs/
```

These locations should not be used as authoritative documentation. Promote only durable evidence into `docs/evidence/` with a short README explaining why it is retained.

## Evidence retention

Evidence should be kept only when it is tied to a decision, release, incident, regression, or acceptance criterion. Each retained evidence directory should include:

```markdown
Status: active | archived | superseded
Related commit:
Related issue/PR:
Reason retained:
Review/delete after:
```

## Plan lifecycle

Every plan in `docs/plans/` should start with metadata:

```markdown
---
status: active | done | superseded | abandoned
owner: xdcjie
created: YYYY-MM-DD
last_reviewed: YYYY-MM-DD
superseded_by:
related_pr:
---
```

Use `active` only for work that is still expected to guide current implementation. Move completed, replaced, or stale plans to an archive folder or mark them clearly at the top.

## Test lifecycle

Tests should not be deleted just because they are old. Classify them first:

- Keep as `unit` if fast and deterministic.
- Keep as `regression` if it captures a real historical bug.
- Move to `integration` or `e2e` if it needs external services or a running stack.
- Move one-off/manual verification scripts out of the default test path or mark them explicitly.
- Delete only when the behavior is obsolete and no longer part of the supported system.

## Large files

Treat unusually large Markdown, JSON, CSV, log, or snapshot files as suspicious until classified. The audit script flags Markdown files larger than 128 KiB by default.
